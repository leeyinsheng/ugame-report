# Phase 0 — 需求釐清：週統計 / 月統計加入 GGR 指標

## 我對目標的理解

**最終要達成的結果：**
在週統計（v1.8）與月統計（v1.7）的表格中增加 **GGR（平台毛利潤）** 欄位，與現有的「投注總額 / 淨利潤NGR / 有效打碼 / 充值總額 / 提現總額 / 活躍會員 / 新增註冊 / 首充會員」並列。

**目標對象：**
內部運營團隊，透過週/月層級掌握平台毛損益。

**要解決的問題：**
當前週/月統計只有 `NGR = GGR - 彩金 - 返水`，但缺少 `GGR = 投注 - 派彩` 這個基礎毛損益指標，導致無法直接從週/月報表看到平台打水結果。

**GGR 已在每日看板中顯示**（`rev.GGR`），公式與平台輸贏一致，僅需在週/月彙總中補上。

## 我的假設

**技術方案：**
- 在 `_monthly_stats()` 與 `_weekly_stats()` 的彙總 dict 內新增 `ggr` 暫存，與 `ngr` 同樣從 `(投注額 - 派彩額)` 計算
- 在 `index.html` 的 `MONTHLY_COLS` 陣列中新增一欄 `['GGR','m']`，放在「投注總額」之後、「淨利潤NGR」之前
- GGR 也會跟著既有環比邏輯自動比對前月/前週

**範圍邊界：**
- 僅在週統計與月統計表格中新增 GGR 欄位，不影響每日看板（已存在）
- 不影響活動彩金、留存等區塊
- 不修改後端 API 結構（僅在 `_meta.monthly` / `_meta.weekly` 中新增 key）

**隱含假設：**
- GGR 公式與現有每日一致：`GGR = 投注金額 - 派彩金額`（正數 = 平台贏）
- 環比計算方式與其他指標相同：`(本月 - 上月) / |上月| * 100`
- 不需要在週/月層級顯示「GGR 殺率」等其他衍生指標

## 盲點 / 模糊點

1. **欄位位置**：GGR 應放在「投注總額」之後、「淨利潤 NGR」之前，還是放在表格最末？請確認你偏好的順序。
2. **環比計算**：是否要對 GGR 計算環比（與其他指標一致）？合理但先確認。
3. **命名**：表格表頭用「平台毛利GGR」還是簡寫「GGR」？每日看板用「平台毛利润 GGR」。
4. **是否需要同時在週/月層級加入 GGR 殺率（GGR / 有效打碼）**？暫時不納入，等你指示。
