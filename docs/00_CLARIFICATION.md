# v1.15 需求釐清 — 頁面載入速度優化

## 我對目標的理解

伺服器重啟（或首次部署）後，第一次請求 `/api/summary` 需從 OSS 下載大量 CSV 並全量彙總，導致：
- **載入時間過久**：14+ 天 × 每日 7+ 個 CSV = 100+ 檔案，下載 + 解析 + 聚合耗時數十秒
- **OOM 風險**：3.5G RAM 機器上全量載入峰值達 3.2G，被 kernel OOM killer 殺死
- **使用者體驗差**：瀏覽器乾等，無任何進度回饋

## 我的假設

**架構現狀：**
- `aggregate.aggregate()` 一次載入所有 CSV 到記憶體 dict，再逐日聚合
- 磁碟快取 `state.pkl` 放在 `/tmp/ugame-dashboard-cache/`（`PrivateTmp=true` → 重啟即清空）
- OSS 上有 14+ 天資料，每日包含注單明細、充提、帳變、會員快照、活動報表

**可行的優化方向（由簡到繁）：**

### 方向 A — 避免全量載入（中優先級）
- **A1：惰性載入** — `/api/summary` 只回傳每日看板（日資料最輕量），月/週統計另開 `/api/monthly`、`/api/weekly` 端點，前端切分頁才觸發
- **A2：背景預熱** — 啟動後 thread 非同步載入資料並寫入快取，讓第一個使用者請求時快取已就緒

### 方向 B — 降低記憶體峰值（高優先級）
- **B1：分區聚合** — 改用 SQLite 做中間層，逐檔匯入 sqlite → 再下 SQL 聚合（memory 只留 sqlite connection）
- **B2：增量 merge** — 只對新檔案做 parse，舊資料直接從 pickle 反序列化（已實作 `only_keys` + `intermediate` 機制，但 intermediate dict 仍含所有原始資料）
- **B3：加 swap** — ECS 加 2G swap file，防止 OOM，治標

### 方向 C — 使用者體驗（低優先級）
- **C1：載入進度條** — SSE 或 polling 顯示下載/解析進度
- **C2：骨架屏** — 先 render 空表格，資料到齊再填入

## 盲點 / 模糊點

1. **實際 OSS 資料量有多大？** — 每天幾個 CSV、各多大 size？這影響方案選擇
2. **可接受的載入時間？** — 目標是 5 秒、10 秒，還是「不 OOM 就行」？
3. **重啟頻率？** — 如果很少重啟（快取長期有效），投資複雜優化的 ROII 較低
4. **swap 是否可行？** — ECS 實例規格？是否允許加 swap？
5. **前端改動範圍？** — 是否允許分割 API / 加 UI 提示？還是只想改後端？
6. **預算考量？** — 換更大規格 ECS 是最簡單的方案
